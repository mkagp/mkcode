# Persisted and protocol identifier compatibility inventory

## Rule

Future branding work must distinguish **user-visible display names** from
**persisted, executable, package, and protocol identifiers**.

Display text and new assets may change earlier. Persisted or protocol-level
identifiers require an explicit migration plan using one or more of:

- fallback reads from the inherited identifier;
- dual-read/new-write behavior for a defined compatibility period;
- aliases for environment variables, schemes, packages, or endpoints;
- one-time, idempotent data migration with backup and rollback;
- versioned protocol/schema negotiation; or
- deliberate retention when migration cost exceeds the benefit.

Do not globally replace `t3`, `t3code`, `@t3tools`, `.t3`, or `T3CODE_*`.

## Migration guardrails

1. Inventory readers, writers, tests, docs, build filters, and external consumers.
2. Define source and destination identifiers plus conflict precedence.
3. Back up persisted state and make migration idempotent.
4. Read legacy and new formats before switching the writer.
5. Verify successful write/decode before deleting old data.
6. Keep rollback able to open the pre-migration state.
7. Never rewrite shipped SQLite migrations or Git history for branding.
8. Treat credentials, DPoP keys, cookies, and pairing links as security migrations,
   not string replacements.
9. Record the compatibility window and removal criteria in an ADR or migration
   plan before implementation.
